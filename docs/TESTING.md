## 1단계. 체크리스트

### 1-A. 요구사항을 모두 만족하는가? (요구사항 추적표)

| 요구사항 (슬라이드) | 구현 위치 | 확인 |
|---|---|---|
| 로그인 페이지 | 첫 화면 로그인 탭 | ☑ |
| 회원가입 페이지 | 첫 화면 회원가입 탭 | ☑ |
| 아이디 중복 불가 | `users.name UNIQUE` + 가입 시 검사 | ☑ |
| 사용자 프로필 확인(조회) | 상품 상세의 판매자 이름 클릭 → 프로필 | ☑ |
| 마이페이지: 소개글 업데이트 | 마이 탭 → 소개글 저장 | ☑ |
| 마이페이지: 비밀번호 업데이트 | 마이 탭 → 비밀번호 변경 | ☑ |
| 유저 정보 DB 관리 | `users` 테이블 | ☑ |
| 상품 등록 페이지 | 판매 탭 | ☑ |
| 내 상품 확인·관리 | 마이 탭 → 내가 올린 상품, 삭제 | ☑ |
| 상품명·가격·사진 표시 | 피드 카드 + 상세 페이지 | ☑ |
| 등록 상품은 누구나 조회 | 모든 유저에게 전체 피드 노출 | ☑ |
| 상품 정보 DB 관리 | `products` 테이블 + `uploads/` | ☑ |
| 이름 표시 + 클릭 시 상세 페이지 | 피드 → 상세 | ☑ |
| 1대1 채팅 | 상품별 구매자↔판매자 대화방 | ☑ |
| 전체 유저 채팅(실시간) | 채팅 탭 → 전체 채팅 (폴링 갱신) | ☑ |
| 신고 기능 + 사유 작성 | 상세/채팅의 🚩 신고 | ☑ |
| 일정 횟수 이상 신고된 상품 차단 | 서로 다른 신고자 N명(기본 3) → 자동 차단 | ☑ |
| 일정 횟수 이상 신고된 유저 휴면 전환 | 동일 임계값 → `dormant=1`, 로그인 차단 | ☑ |
| 불량 상품 삭제 / 불량 유저 제재 (관리자) | 관리자 콘솔: 차단·복구·휴면 해제 | ☑ |

> 임계값은 `.env`의 `REPORT_AUTO_LIMIT`으로 조정 (기본 3). 한 명이 여러 번 신고해도 1명으로 계산합니다(신고 도배 악용 방지).

### 1-C. 보안 요소가 제대로 구현되어 있는가?

| # | 점검 항목 | 확인 방법 |
|---|---|---|
| S1 | 비밀번호 평문 저장 금지 | `data/dasijang.db`를 열어 `users.pw_hash`가 `$2a$...` 해시인지 확인 |
| S2 | 로그인 없이 API 접근 불가 | 토큰 없이 `curl localhost:3000/api/products` → 401 |
| S3 | 일반 유저의 관리자 API 접근 불가 | 일반 토큰으로 `/api/admin/summary` → 403 |
| S4 | 남의 대화방 열람 불가 | 제3자 토큰으로 대화방 조회 → 403 |
| S5 | 잔액 마이너스 불가 | 잔액 초과 송금/구매 → 거절 (음수·0원도 거절) |
| S6 | 결제 금액 변조 차단 | 승인 요청의 amount가 서버 기록과 다르면 거절 |
| S7 | 결제 중복 승인 차단 | 같은 주문번호 재승인 → 잔액 안 늘어남 (멱등) |
| S8 | XSS 차단 | 상품명에 `<script>alert(1)</script>` 등록 → 글자로만 표시, 팝업 없음 |
| S9 | SQL 인젝션 차단 | 검색창에 `' OR 1=1 --` 입력 → 오류 없이 그냥 검색어로 처리 |
| S10 | 파일 업로드 제한 | 이미지 아닌 파일/5MB 초과 → 거절 |
| S11 | 차단·휴면 계정 격리 | 로그인 거절 + 상품 피드 제외 + 송금 수신 불가 |
| S12 | 로그인 브루트 포스 방어 | **IP 기준** 제한: 한 IP가 10분 내 10회 실패 → 그 IP 로그인 10분 429. 계정을 잠그지 않아 계정 잠금 DoS 불가 |
| S13 | 업로드 파일 위장 차단 | 파일 내용(매직 바이트) 검증, 확장자는 서버가 부여, nosniff 헤더 (진단서 §1) |
| S14 | 토큰 탈취 방어 | JWT를 httpOnly+SameSite 쿠키로 보관 — JS로 접근 불가 (진단서 §2) |
| S15 | 목록 조회 페이징 | 전 목록 API에 LIMIT 100 + page 파라미터 (진단서 §3) |
| S16 | 도배 방지 | 유저당 1일 상품 등록 30개 제한, 삭제분 포함 집계 (진단서 §4) |
| S17 | 피싱 링크 경고 | 채팅·상품 설명에 URL 감지 시 경고 문구 상시 표시 (진단서 §5) |
| S18 | 금액 입력 방어 | 지수표기(1e300)·문자열·소수·과대 금액 거절, 1억원 상한 (2차 진단) |
| S19 | 페이징 크래시 방어 | 거대 page 값에도 500 없이 정상 응답 (OFFSET 상한) (2차 진단) |
| S20 | 업로드 용량 초과 처리 | 5MB 초과 시 500이 아닌 413으로 응답 (2차 진단) |
| S21 | 전체채팅 도배 방지 | 유저당 10초 8건·최소 1.5초 간격 제한 (2차 진단) |
| S22 | 계정 잠금 DoS 방지 | 제한 대상이 계정이 아닌 요청 IP → 공격자가 타인 계정을 잠글 수 없음 (2차 진단) |
| S23 | 모의결제 격리 | mock-confirm 라우트는 mock 모드에서만 등록 → 운영(toss)에선 404, 공짜충전 불가 (정적검토) |
| S24 | 신고 오남용 축소 | 자기/본인상품/중복/관리자 신고 차단 (정적검토) |
| S25 | CSRF 방어 | 쿠키 인증 상태변경 요청에 X-CSRF-Token 헤더 검증(double-submit) + SameSite=Strict (정적검토) |
| S26 | JWT 알고리즘 고정 | HS256만 허용 → alg:none·알고리즘 혼동 공격 차단 (정적검토) |
| S27 | 운영 시크릿 강제 | NODE_ENV=production에서 기본 JWT_SECRET이면 기동 거부 (정적검토) |
| S28 | 전역 API 레이트 리밋 | 모든 API에 IP당 분당 300건 상한(429) → 검색 등 애플리케이션 계층 DoS 완화 (시큐어코딩 보강) |
| S29 | 보안 HTTP 헤더 | CSP·X-Frame-Options(DENY)·Referrer-Policy·COOP·(HTTPS 시)HSTS, x-powered-by 제거 (시큐어코딩 보강) |
| S30 | 에러 정보 노출 차단 | 500 오류는 일반 문구만 반환, 스택·SQL·경로 미노출. 잘못된/거대 요청은 400·413 (시큐어코딩 보강) |
| S31 | 요청 본문 크기 제한 | JSON 바디 1MB 상한 → 대용량 페이로드 메모리 남용 차단 (시큐어코딩 보강) |

## 2단계. 실제 테스트

### 자동 테스트 (25개 케이스 일괄 실행)

```bash
# ① 깨끗한 DB로 시작 (기존 데이터가 있으면 백업 후)
rm data/dasijang.db*
npm start                      # 터미널 1에서 서버 실행

# ② 다른 터미널에서
bash tests/e2e.sh              # → "결과: 통과 49 / 실패 0" 이 나와야 함
```

이 스크립트는 위 체크리스트의 F/S 항목 대부분을 API 레벨에서 자동 검증합니다.
코드를 고칠 때마다 실행해서 **기존 기능이 깨지지 않았는지(회귀 테스트)** 확인하세요.

### 수동 테스트 (화면 확인)

자동 테스트가 못 보는 "화면"을 사람이 확인합니다:
1. 일반 창에 유저 A, **시크릿 창**(Ctrl+Shift+N)에 유저 B 로그인
2. F5~F12를 화면에서 직접 수행 — 특히 채팅이 2~3초 안에 상대 창에 뜨는지
3. 휴대폰 크기로 창을 줄여 레이아웃이 깨지지 않는지
4. S8(XSS)은 반드시 화면에서: 팝업이 뜨면 실패

## 3단계. 유지보수

### 운영 루틴

- **백업**: `data/dasijang.db` 파일과 `uploads/` 폴더만 복사하면 전체 백업 (`cp -r data uploads ~/백업/`)
- **로그 확인**: 서버 터미널에 500 오류가 찍히면 그 줄이 원인 파일·행 번호를 알려줌
- **의존성 점검**: 가끔 `npm audit` 실행 → 취약점 발견 시 `npm audit fix`
- **비밀값 관리**: `.env`는 절대 공유/커밋 금지 (`.gitignore`에 이미 등록됨)

### 문제 발견 시: 적절한 단계로 돌아가기 (수업 자료의 회귀 원칙)

| 발견한 문제 | 돌아갈 단계 |
|---|---|
| "이 기능이 애초에 필요 없었네 / 다르게 동작해야 하네" | **요구사항 분석**부터 다시 (설계 문서 수정 → 구현) |
| "기능은 맞는데 구조가 이상하다" (예: 돈 계산이 여러 곳에 흩어짐) | **설계** 단계 (docs/ARCHITECTURE.md 갱신 후 리팩토링) |
| "동작이 기대와 다르다" (버그) | **구현** 단계 (수정 → `bash tests/e2e.sh`로 회귀 확인) |
| "정상인데 불편하다" (예: 채팅이 느리게 느껴짐) | **개선 백로그**에 기록 후 우선순위 정해 반영 |

### 수정 → 검증 사이클 (버그를 고칠 때마다)

1. 문제를 **재현하는 테스트 케이스**를 먼저 `tests/e2e.sh`에 추가 (실패하는 상태 확인)
2. 코드를 수정
3. `bash tests/e2e.sh` 전체 실행 — 새 케이스가 통과하고 **기존 34개도 여전히 통과**해야 완료

> 실제 사례: 비밀번호 입력이 안 되던 버그(onkeydown이 false를 반환해 키 입력을 취소)도
> 이 사이클로 처리했습니다 — 재현 → 원인 파악 → 수정 → 전체 회귀 테스트.

### 확장 시 보안 고려사항 (단일 서버 → 다중 서버)

현재 여러 방어 장치(로그인 브루트포스 카운터[S12], 전체채팅 도배 제한[S21],
전역 API 레이트 리밋[S28])는 **각 서버 프로세스의 메모리**에 상태를 둔다.
이는 단일 서버에서는 충분하지만, 다음 두 상황에서 한계가 있다.

1. **서버 재시작 시 초기화**: 프로세스가 재시작되면 카운터가 리셋된다.
   공격자가 서버 크래시/재배포를 유발할 수 있다면 방어 카운터를 우회할 여지가 생긴다.
2. **서버 다중화 시 분산**: 부하 분산을 위해 서버를 여러 대로 늘리면(로드밸런서 뒤),
   카운터가 서버마다 따로 존재한다. 공격자가 "서버 A에서 몇 번, 서버 B에서 몇 번" 식으로
   요청을 분산시키면 각 서버 기준으로는 임계값에 못 미쳐 제한을 우회할 수 있다.

**대안(다중화 시 필수): 공유 저장소(Redis)로 전환.**
카운터를 각 서버 메모리가 아니라 모든 서버가 함께 보는 Redis에 두면,
"서버 A 3건 + 서버 B 2건 = 합산 5건"으로 정확히 집계되고 재시작에도 유지된다.
`ioredis` 같은 클라이언트로 `INCR`+`EXPIRE`를 쓰거나 `rate-limiter-flexible` 라이브러리로
교체하면 되며, 애플리케이션 로직(어디서 카운트를 올리고 검사하는지)은 그대로 두고
저장 위치만 메모리 → Redis로 바꾸면 된다.

> 본 과제 범위(단일 서버 데모)에서는 메모리 기반으로 충분하며, 위 한계와 대안을
> 명시하는 것으로 갈음한다. 실서비스 다중화 단계에서 Redis 전환을 우선 과제로 둔다.

### 개선 백로그 (다음 단계 후보)

- **회원가입 시 전화번호(SMS) 인증 — 신고 남용 방지 (우선순위 높음)**
  - 배경: 현재 자동 제재는 "서로 다른 신고자 3명"을 기준으로 한다. 그런데 가입에 아무 제약이 없어,
    한 사람이 여러 계정을 대량 생성(다중 계정)해 특정 상품·유저를 3회 이상 신고하면
    **정상 판매자를 부당하게 차단·휴면시키는 신고 남용**이 가능하다.
  - 부분 조치 완료: 자기 신고·본인 상품 신고·동일인 중복 신고·관리자 신고를 차단해
    한 계정이 카운트를 부풀리는 경로는 막았다. 다만 **여러 계정 생성** 자체는 아직 못 막는다.
  - 근본 대책(미구현): 회원가입에 휴대폰 번호 SMS 인증을 도입해 **1인 1계정(번호당 1계정)**을 강제한다.
    이렇게 하면 다중 계정 생성 비용이 크게 올라가 신고 남용·도배·어뷰징이 함께 억제된다.
  - 구현 개요: 가입 폼에 전화번호 입력 → 문자 인증번호 발송(예: NHN Cloud/CoolSMS/Twilio 등 SMS API)
    → 인증 성공 시에만 계정 생성. `users` 테이블에 `phone`(유니크) 컬럼 추가로 번호 중복 가입 차단.
    인증번호는 5분 만료·시도 횟수 제한을 둔다.
